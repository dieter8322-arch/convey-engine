# Open Source File Header

Use this skill whenever you create a new source file for this repository or when the user asks to normalize file headers.

## Goal

Keep file-level copyright and licensing metadata machine-readable, standard, and compatible with REUSE / SPDX practices.

Do not invent custom header formats. Do not use `Created by`, `Author:`, system usernames, or local hidden config values as the default file header strategy.

## Default Rule For First-Party Source Files

For new first-party source files, add these two tags near the top of the file using the correct comment syntax for that language:

```go
// SPDX-FileCopyrightText: 2026 dieter8322-arch
// SPDX-License-Identifier: MIT
```

## Rules

1. New first-party source files should use `SPDX-FileCopyrightText` and `SPDX-License-Identifier`.
2. Use the repository's current first-party copyright holder string: `2026 dieter8322-arch`.
3. Use `MIT` as the default SPDX license identifier for first-party files.
4. Preserve existing third-party copyright and license notices when copying, vendoring, or adapting upstream code.
5. If a file has multiple copyright holders, add multiple `SPDX-FileCopyrightText` lines.
6. Do not add `Created by`, `Author:`, usernames, emails, or local machine identities unless the user explicitly asks.
7. Generated files should keep the generated notice above the SPDX tags.
8. If a file cannot or should not contain inline headers, use repository-level REUSE metadata such as `REUSE.toml`.

## Default Header Forms

### Go / TypeScript / JavaScript

```ts
// SPDX-FileCopyrightText: 2026 dieter8322-arch
// SPDX-License-Identifier: MIT
```

### CSS

```css
/* SPDX-FileCopyrightText: 2026 dieter8322-arch */
/* SPDX-License-Identifier: MIT */
```

### Shell / Python / Dockerfile / Makefile

```sh
# SPDX-FileCopyrightText: 2026 dieter8322-arch
# SPDX-License-Identifier: MIT
```

### SQL / Lua

```sql
-- SPDX-FileCopyrightText: 2026 dieter8322-arch
-- SPDX-License-Identifier: MIT
```

### Generated Source File

```go
// Code generated by <tool>; DO NOT EDIT.
// SPDX-FileCopyrightText: 2026 dieter8322-arch
// SPDX-License-Identifier: MIT
```

Replace `<tool>` with the actual generator name when known.

## Third-Party And Mixed-Origin Files

- If the file comes from a third party, preserve upstream notices.
- If you substantially extend a third-party file and policy requires adding your own notice, add it as an additional `SPDX-FileCopyrightText` line instead of replacing upstream metadata.
- If you copy only a snippet, use snippet-level SPDX annotations when needed.

## When Not To Use Inline Headers

- Markdown docs when the repository already covers them via `REUSE.toml`
- JSON / YAML / TOML config when inline comments are undesirable
- Lockfiles
- Binary assets

## Validation

- Confirm the file header uses valid comment syntax for that file type.
- Confirm first-party source files include both SPDX tags.
- Confirm generated files keep the generated notice above the SPDX tags.
- Confirm copied or adapted third-party files keep upstream metadata.
- Confirm no custom `Created by` style header remains unless explicitly requested.
